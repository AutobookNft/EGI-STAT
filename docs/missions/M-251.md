# M-251 — Rebuild preserva tabelle legacy (righe nette) + totali tabella

> Report tecnico (FASE 6a). Esteso: M-251_FULL.md.

- **Organo**: EGI-STAT. Trigger=2.
- **Sintomo**: tabella "Ore per Progetto" → righe nette crollate/azzerate (EGI 717.476→20.957; NATAN_LOC/EGI-HUB/altri → 0), ore ~dimezzate.
- **Causa (misurata, DeepDebug)**: regressione del fix atomico M-250. Le tabelle `legacy_production`/`legacy_repo_day` (massa storica delle righe nette, popolate da `ingest_legacy_production.py`, NON da `aggregate()`) non sono in `DROP_TABLES`. Il vecchio rebuild faceva DROP+CREATE sullo stesso file → le legacy sopravvivevano. Il rebuild atomico M-250 crea un DB **nuovo da zero** → legacy **perse** → righe nette per repo a 0.
- **Fix**: `aggregate()` ora parte da una **copia** del DB esistente (`shutil.copy2`) prima del rebuild; `create_schema` droppa/ricrea solo le tabelle mission/time → le legacy sopravvivono. Re-ingest (`ingest_legacy_production.py`) ripristina i dati persi.
- **Recupero verificato**: EGI lines_net 741.395, NATAN_LOC 222.071, TOTALE righe nette 2.087.976.
- **Aggiunta**: riga **TOTALE** in fondo alla tabella "Ore per Progetto" (richiesta CEO) — somma ore/min/righe nette.
- **Test**: `tests/m_251_preserve_legacy_test.py` (la legacy sopravvive al rebuild) 1/1; regressione 21/21.
- **Residuo noto (secondario)**: alcune mission schema-EN (`M-EGI-NNN`) hanno `stats=0` perché il re-enrich `--force` cercava `M-EGI-249` ma i commit taggano `M-249` (grep a vuoto). Recuperabile via git restore dei registry (cross-repo, approvazione CEO). Non è la massa (che veniva da legacy, ora recuperata).
