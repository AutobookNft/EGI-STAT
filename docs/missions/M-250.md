# M-250 — Rebuild stats.db ATOMICO: basta dashboard a zero (RACE)

> Report tecnico (FASE 6a). Esteso: M-250_FULL.md. Regressione trovata da DeepDebug, riprodotta.

- **Organo**: EGI-STAT. Trigger=2.
- **Sintomo**: dashboard con metriche mission-derived (PI, CL, Mission Chiuse, Commit Pesati, Mission/giorno, daily snapshot) a ZERO; righe/file/ore popolati. Funzionava ~2h prima. L'API `curl` però restituiva dati corretti (W23=168) → contraddizione.
- **VERA causa (DeepDebug, riprodotta — non dedotta)**: M-247 chiama `ensure_fresh` in `stats_v2._connect()` → rebuild on-read; `aggregate()` faceva `DROP+CREATE` sul DB **servito**. Dopo il re-enrich globale TUTTI i registry erano più recenti del DB → `is_stale=True` a ogni richiesta → rebuild continuo. Sotto 2 worker Flask, una richiesta leggeva il DB **a metà ricostruzione** → `missions=0`. Le tabelle legacy (righe/file/ore) sopravvivevano in altra fase → split selettivo esatto osservato.
- **Prova riprodotta**: DB reso stale + 10 richieste parallele → **7/10 tornavano 0** (alcune 114 settimane = parziale).
- **Fix**: `aggregate()` costruisce su `<db>.building.<pid>` e fa **`os.replace()` atomico** → i lettori vedono SEMPRE un DB completo (vecchio o nuovo, mai a metà). Guard: se 0 registry scoperti, non sovrascrive il DB valido.
- **Verifica**: dopo fix, stale + 10 parallele → **10/10 corrette (168)**. Test `tests/m_250_atomic_rebuild_test.py` (lettore concorrente non vede mai DB vuoto) 1/1; regressione 20/20. Servizio riavviato.
